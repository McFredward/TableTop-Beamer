PASS signals in start.log on Win32:

1. Launch banner (one line, ~5s after start.bat):
   `[ssr-host] launching headless=new on Win32 (userDataDir=C:\Users\...\Temp\ttb-ssr-NNNN-TTTTTTTT)`

2. Verdict line (one line, ~10-30s after start.bat — depends on Chrome startup time):
   `[ssr-host] win32 verdict: OK browserConnected=true producerIds=[<one-or-more-csv-ids>]`

3. Optional args dump (only if `set SSR_LOG_LAUNCH_ARGS=1` before start.bat):
   `[ssr-host] launch args (win32): --no-sandbox --autoplay-policy=no-user-gesture-required --use-gl=angle ...`

FAIL signals (anything other than the above):
- `[ssr-host] launching headless=false on Win32 ...` → operator deliberately used escape hatch (or accidentally has SSR_WIN_HEADLESS=0 set — note for operator's debug section)
- `[ssr-host] win32 verdict: FAILED <reason>` → publisher injection failed; operator copies the reason into 47-UAT-RESULTS.md gap section
- Missing both lines after 60s → boot stalled before launchBrowser was even called; operator pastes the last 50 lines of start.log

Cleanup-matrix probe commands (Win11 cmd or PowerShell):
- Before Ctrl+C:  `tasklist | findstr /i "node.exe chrome.exe mediasoup"`
- After Ctrl+C (wait 5s): same command — expected output: empty
- Same flow with the cmd-window-X-button instead of Ctrl+C (Q5 row 2)
- Same flow with Task-Manager-End-Task on cmd.exe (Q5 row 3)
- Same flow with Task-Manager-End-Task on node.exe (Q5 row 4 — acceptable: chrome.exe lingers <5s, then dies on its own per D-05)
