Three new log lines (all gated by `process.platform === "win32"` except #3):

(1) Launch banner — emitted ONCE inside launchBrowser, right before the
`return launcher(...)` call. Replaces the smaller Wave-2 log:
    `[ssr-host] launching headless=new on Win32 (userDataDir=<tmppath>)`
    or
    `[ssr-host] launching headless=false on Win32 (userDataDir=<tmppath>, SSR_WIN_HEADLESS=0)`

Format (template literal):
```js
logger.info(`[ssr-host] launching headless=${useHeadlessNew ? "new" : "false"} on Win32 ` +
            `(userDataDir=${winUserDataDir ?? "<none>"}${useHeadlessNew ? "" : ", SSR_WIN_HEADLESS=0"})`);
```

(2) Verdict line — emitted ONCE after `status.browserConnected = true` and
the in-page publisher has either succeeded or failed. Two forms:
    `[ssr-host] win32 verdict: OK browserConnected=true producerIds=[<csv>]`
    `[ssr-host] win32 verdict: FAILED <reason>`

Location: in the start/connect flow, right after the `injectInPagePublisher`
try/catch around line 908. The catch sets status.lastError; the verdict
must reflect that:
```js
if (process.platform === "win32") {
  if (status.lastError) {
    logger.info(`[ssr-host] win32 verdict: FAILED ${status.lastError}`);
  } else {
    logger.info(`[ssr-host] win32 verdict: OK browserConnected=${status.browserConnected} producerIds=[${(status.producerIds ?? []).join(",")}]`);
  }
}
```

(3) Args dump — emitted ONCE inside launchBrowser, gated by env var
SSR_LOG_LAUNCH_ARGS=1 (no platform gate — useful for Linux debugging too,
but the operator's docs only mention it for Windows):
```js
const chromiumArgs = buildChromiumLaunchArgs({ ... });
if (process.env.SSR_LOG_LAUNCH_ARGS === "1") {
  logger.info(`[ssr-host] launch args (${process.platform}): ${chromiumArgs.join(" ")}`);
}
return launcher({ ..., args: chromiumArgs, ... });
```

Linux path emits NO new logs at boot (the win32 gates ensure that). Args dump
fires only when explicitly requested via env var on EITHER platform.
