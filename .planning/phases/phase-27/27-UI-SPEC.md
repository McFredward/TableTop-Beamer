## Spacing Scale

Standard 8-point grid used throughout. No new spacing tokens are introduced.

| Token | Value | Usage in Phase 27 |
|-------|-------|-------------------|
| xs | 4px | Dirty-dot gap from profile name; chip internal padding-top/bottom |
| sm | 8px | Chip internal padding-left/right; toolbar gap between items |
| md | 16px | Toolbar horizontal padding |
| lg | 24px | Minimum squish-bar length (perpendicular) |
| xl | 32px | Squish-bar length along the outer edge |

Phase 27's new spacing scale is 4-px-grid-aligned (xs=4, sm=8, md=16, lg=24, xl=32).

### Inherited Geometry (Not Part of New Spacing Scale)

The following values are NOT new spacing tokens — they are immutable inherited geometry from existing align-mode code that Phase 27 reuses verbatim. They are documented here for completeness so implementors do not introduce parallel non-grid values.

| Value | Source | Why it must stay exactly this value |
|-------|--------|-------------------------------------|
| **30px** squish-bar offset from outer edge midpoint | `runtime-projection-handle-ui.js` — existing rotate handle `offX: ±30, offY: ±30` | Must match exactly to preserve visual symmetry with rotate handles. Changing to 32px would break the visual alignment between squish bars and rotate handles. |
| **18px** intersection handle touch target | Pre-existing immutable handle DOM | Phase 27 does not modify the handle DOM. This is the current rendered diameter; any change would require modifying code outside Phase 27 scope. |

Line hit-test thresholds (locked in D-10, not spacing tokens): 6px for line-only hits; 10px for intersection hits.

## Component Specifications

### B3/B4 — Profile Chip + Dirty-Flag Toolbar

**Location:** Fixed overlay, top-center of the viewport, visible only when align-mode is active on `/output/`. Stacks below the diagnostic status chip (`z-index: 9999`) but above the line canvas (`z-index: 9997`). Use `z-index: 9998`.

**Structure:**

```
[ ● Profile name ]  [ Save profile ]  [ Save as new… ]  [ Discard ]
```

All elements are in a single horizontal flex row: `display:flex; align-items:center; gap:8px; padding:6px 12px; border-radius:999px; background:rgba(14,22,34,0.92); border:1px solid var(--c-border); backdrop-filter:blur(6px); box-shadow:0 2px 8px rgba(0,0,0,0.5)`.

**Profile chip pill** (the `[ ● Name ]` portion):
- Background: `rgba(14,22,34,0.92)` — same as toolbar (they share the container)
- Profile name text: `var(--c-text)` when clean; `var(--c-warn)` when dirty
- Name font: 12px / 600 / Space Grotesk
- Dirty dot `●`: 10px, color `var(--c-warn)`, `margin-right:4px`, only visible when dirty

**Clean state (no unsaved changes):**
- No dirty dot
- Profile name in `var(--c-text)`
- When no profile loaded: display text "Unsaved" in `var(--c-text-2)` (muted)
- Save profile / Save as new / Discard: all three buttons visible but Discard is `opacity:0.35; pointer-events:none` (nothing to discard)

**Dirty state (unsaved changes exist):**
- Dirty dot `●` visible, `var(--c-warn)`
- Profile name in `var(--c-warn)`
- Chip pill border: `1px solid rgba(245,181,68,0.5)` (dark) / `rgba(185,128,31,0.5)` (light)
- Save profile button: primary styling — `background:var(--c-accent); color:var(--c-accent-fg); border:none`
- Save profile button hover: `background:var(--c-accent-2)` (darker teal on hover)
- Save as new button: ghost styling — `background:transparent; color:var(--c-text); border:1px solid var(--c-border-str)`
- Discard button: danger-tinted text — `background:transparent; color:var(--c-danger); border:1px solid rgba(255,91,91,0.35)`
- Discard hover: `background:var(--c-danger-bg); border-color:var(--c-danger)`

**When no profile is loaded (name = "Unsaved") and dirty:**
- "Save profile" acts as "Save as new…" — triggers the name-input modal directly
- Button label stays "Save profile" (user expectation: saving = committing changes)

**Button sizing:**
- min-height: 28px (compact — this is an overlay on the projector canvas, not a full dashboard button)
- padding: 4px 10px
- border-radius: 6px
- font: 12px / 600 / Space Grotesk
- transition: `background 80ms ease, border-color 80ms ease`

**Focus rings:** `outline:2px solid var(--c-accent); outline-offset:2px` — inherits from global button `:focus-visible` rule.

**Keyboard:**
- Tab order: profile chip (read-only display) → Save profile → Save as new → Discard
- Escape inside toolbar: no action (Escape is owned by the existing grid reset; do not intercept)

### B7/B8/D-10 — Align Mode Right-Click Context Menu

Reuse `.board-context-menu` + `.board-context-menu-item` CSS classes exactly. The existing align-mode `showContextMenu()` function already uses these classes; Phase 27 replaces the item logic only, not the styling or positioning code.

**Menu structure by hit-test result:**

| Hit-test target | Menu items shown |
|----------------|-----------------|
| Empty canvas (no line, no intersection within threshold) | "Add horizontal line here" · "Add vertical line here" |
| On an inner line only (within 6px of line, not within 10px of any intersection) | "Delete this line" · "Add line through this point" |
| On an intersection (within 10px of a grid point) — inner lines only | "Delete vertical line" · "Delete horizontal line" · "Add line through this point" |
| On an intersection that is ON an outer (boundary) line | Same as intersection, but the delete option for the outer-line side is hidden (not greyed — removed entirely) |

**Item labels — exact copy:**
- `"Add horizontal line here"` (empty canvas)
- `"Add vertical line here"` (empty canvas)
- `"Delete this line"` (on a single inner line)
- `"Add line through this point"` (on a line or intersection — inserts perpendicular line at click coordinate)
- `"Delete vertical line"` (on intersection, vertical line is inner)
- `"Delete horizontal line"` (on intersection, horizontal line is inner)

**Destructive items styling:**
- "Delete this line", "Delete vertical line", "Delete horizontal line" use `color:var(--c-danger)` on the button text — no background change, just the text color shifts to signal destructive intent
- No confirmation step (D-11 — deletion is undoable via Ctrl+Z)

**Positioning + dismiss:** Identical to existing `showContextMenu()` — `position:fixed`, `requestAnimationFrame` overflow clamp, dismiss on pointerdown-outside or Escape.

**ARIA:**
- `role="menu"` on the container
- `role="menuitem"` on each button
- `aria-label="Grid line options"` on the container

**Hit-test thresholds (exact, for implementors):**
- Intersection hit: point within 10px (Euclidean) of any `getPoint(row, col)` in screen space
- Line hit: perpendicular distance from cursor to the nearest line segment < 6px, AND cursor is NOT within the 10px intersection radius of either endpoint

### B9/D-12..D-14 — Squish Bars

**Shape:** Slim rounded rectangle, not a circle. This is the key visual distinction from rotate handles (circles with ↻ glyph).

**Dimensions per bar:** 60px wide × 10px tall (for top/bottom bars, horizontal orientation). Left/right bars are rotated: 10px wide × 60px tall.

**Visual style:**

| Property | Value |
|----------|-------|
| Background (resting) | `rgba(50,211,163,0.80)` — teal family, matches `--c-accent` at 80% opacity |
| Border | `1.5px solid rgba(255,255,255,0.85)` |
| Border radius | 5px |
| Cursor | `ns-resize` (top/bottom bars) · `ew-resize` (left/right bars) |
| z-index | 10001 (above rotate handles at 10000) |
| Box shadow | `0 2px 6px rgba(0,0,0,0.5)` (matches rotate handles) |
| user-select | none |
| touch-action | none |

**Glyph / label:** No text glyph. A 2px-wide darker teal line centered across the short axis of the bar (`rgba(5,50,40,0.55)`) acts as a drag affordance indicator — like a tactile ridge. This is drawn as a thin `div` centered inside the bar.

**Hover state:**
- Background: `rgba(50,211,163,1.0)` (fully opaque accent)
- Border: `1.5px solid rgba(255,255,255,1.0)`
- `transform: scale(1.08)` (slight grow, 120ms ease)

**Active/dragging state:**
- Background: `rgba(30,163,120,0.95)` (darker, pressed-in feel)
- `transform: scale(1.0)` (return to normal scale — motion paused during drag)
- cursor: inherited from bar direction (`ns-resize` / `ew-resize`) — no grabbing cursor during squish drag

**Placement:**

Each bar is positioned at the **midpoint of its outer edge**, offset **30px outward** from the outer edge in screen space. This exactly matches the rotate-handle offset pattern (`offX: ±30, offY: ±30` at corners) — see Inherited Geometry note in Spacing Scale.

| Bar | Anchor point | Offset direction |
|-----|-------------|-----------------|
| Top bar | Midpoint of top edge (`getPoint(0, midCol)` x-coord, `getPoint(0,0)` y-coord) | −30px in Y (above the grid) |
| Bottom bar | Midpoint of bottom edge | +30px in Y |
| Left bar | Midpoint of left edge | −30px in X |
| Right bar | Midpoint of right edge | +30px in X |

For non-rectangular (trapezoid) outer shapes (B2 result): the bar sits at the screen-space midpoint of the outer edge (average of the two outer corner screen positions on that side), and the squish operates along the **edge-perpendicular local axis**, not world X/Y.

**Positioning update cadence:** Same as rotate handles — updated every time `positionRotateHandles()` is called (after any drag, resize, or undo).

**DOM injection:** Four `<div>` elements appended to `document.body`, created in `rebuildRotateHandles()` (or a new `rebuildSquishBars()` called from the same site), removed in `removeHandles()`.

**CSS class:** `projection-squish-bar` — add a minimal class rule to `src/styles.css` for the resting state (the JS overrides `left`/`top`/`transform` inline for positioning and orientation, same as rotate handles).

```css
.projection-squish-bar {
  position: fixed;
  cursor: ns-resize; /* overridden per-bar via JS */
  z-index: 10001;
  user-select: none;
  -webkit-user-select: none;
  touch-action: none;
  box-shadow: 0 2px 6px rgba(0,0,0,0.5);
  transition: background 120ms ease, transform 120ms ease;
  pointer-events: auto;
}
```

All other visual properties (width, height, background, border, border-radius) set inline by JS at creation time, matching the pattern of rotate handles.

## Interaction Contracts

### Edge Uniformity (B1) — no visual change needed

After D-01, outer edges use the same interaction code as inner lines. The visual change is: outer lines switch from the existing `rgba(220, 30, 30, 0.7)` red color to `rgba(0, 220, 180, 0.45)` teal — the same as inner lines. The red color was the only remaining visual distinction; removing it signals to the user that they are now equivalent.

**Exception:** The filled quad background `rgba(255, 50, 50, 0.06)` may remain as a subtle board-area indicator — it is not a line color and does not imply edge uniqueness.

### Dirty State Detection

Dirty flag is computed by `runtime-projection-profile-persistence.js` by deep-comparing `grid.srcXs`, `grid.srcYs`, and `grid.points` against the last-saved profile snapshot. Any deviation sets dirty = true. This drives:
1. Profile chip visual (dot + name color)
2. Save/Discard button enable state
3. Broadcast dirty state over `global-config-update` WebSocket channel

### Undo Stack Integration

Every destructive context-menu action (line deletion) must call `pushUndo()` before mutating the grid — same as the existing line-drag operations. The undo count display is not part of Phase 27 scope.
