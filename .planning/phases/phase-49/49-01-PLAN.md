<objective>
Phase 49 — Windows process-supervision hardening. Two operator-reported issues from pre-release testing (2026-05-17):

1. **49-A** — Ctrl+C in an existing PowerShell/cmd session (./start.bat from a real working shell, not a double-click) leaves the server alive AND/OR kills the operator's working shell session. The Phase 47 gap-closure-13 preemptive ancestor-cmd-kill blindly force-kills every cmd/powershell ancestor up to (but not including) explorer.exe — fine for double-click (parent is a wegwerf cmd window), catastrophic for shell-invoked (parent is the operator's working shell).

2. **49-B** — Closing the cmd window via the X-button can orphan node.exe / chrome.exe / mediasoup-worker.exe. The Job Object already has `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE` (0x2000) set on line 567, BUT there is no `SetConsoleCtrlHandler` registered for `CTRL_CLOSE_EVENT`, so the PowerShell cleanup ($cleanup scriptblock) never fires on window-close — no shutdown log line, and we rely entirely on the kernel's hard-kill, which can race with chrome.exe child-process reparenting.

Purpose:
- Fix 49-A by detecting invocation context at boot and selecting between two cleanup paths (fresh-cmd-kill-ancestor vs existing-shell-leave-parent-alone).
- Fix 49-B by registering a SetConsoleCtrlHandler for CTRL_CLOSE_EVENT so the existing $cleanup scriptblock runs BEFORE the kernel hard-kill on window-close (belt + suspenders with the already-set KILL_ON_JOB_CLOSE flag).
- Preserve Phase 47 double-click Ctrl+C behavior exactly.
- Zero changes to start.sh / Linux paths / src/server/*.

Output: Single PLAN with 2 implementation waves + 1 UAT checkpoint modifying only start.ps1.
- Wave 1 (Task 1): 49-B Job Object KILL_ON_JOB_CLOSE flag is verified already set (no code change needed for the flag), and SetConsoleCtrlHandler P/Invoke is added with a CTRL_CLOSE_EVENT delegate that invokes $cleanup. Pure-add change — no existing logic touched.

The new CTRL_CLOSE_EVENT block to add:

```powershell
# Phase 49 (item 49-B, 2026-05-17): register a SetConsoleCtrlHandler for
# CTRL_CLOSE_EVENT so the PowerShell-side $cleanup runs BEFORE the kernel
# hard-kill on window-close. The Job Object already has
# JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE (0x2000, see LimitFlags above), which
# guarantees node/chrome/mediasoup-worker are reaped on parent handle close
# — this handler adds a graceful shutdown window (~5 s on Windows) so we
# get a "[start] Shutting down ..." log line and Stop-ServerProcess gets
# its taskkill /T pass in. Belt + suspenders, not replacement.
#
# Lifetime: $script:CtrlHandlerDelegate keeps the delegate INSTANCE alive
# while Windows still holds the callback pointer. If we let .NET GC the
# delegate, the next CTRL_CLOSE_EVENT calls into freed memory and crashes
# the process. Stored in $script: scope so it survives until natural exit.
try {
  $script:CtrlHandlerDelegate = [TtbJob+ConsoleCtrlDelegate]{
    param([uint32]$ctrlType)
    # 2 = CTRL_CLOSE_EVENT (X-button close, taskkill of cmd, log-off, etc.)
    # 5 = CTRL_LOGOFF_EVENT, 6 = CTRL_SHUTDOWN_EVENT (similar terminal-ish events)
    if ($ctrlType -eq 2 -or $ctrlType -eq 5 -or $ctrlType -eq 6) {
      try { & $cleanup } catch {}
    }
    # Return $false so Windows continues its default handler chain (which
    # will trigger the kernel-level KILL_ON_JOB_CLOSE on the Job Object).
    # If we returned $true Windows would stop dispatching and the
    # KILL_ON_JOB_CLOSE belt would not fire as a safety net.
    return $false
  }
  [void][TtbJob]::SetConsoleCtrlHandler($script:CtrlHandlerDelegate, $true)
} catch {
  Write-Host "[start]    (CTRL_CLOSE_EVENT handler unavailable; relying on Job Object KILL_ON_JOB_CLOSE only)"
}
```

```powershell
# Phase 49 (item 49-A, 2026-05-17): detect invocation context to decide
# whether the Ctrl+C handler is allowed to kill ancestor cmd.exe.
#
# Two operator workflows:
#   (a) Double-click start.bat from Explorer  -> parent is a wegwerf
#       cmd.exe spawned BY explorer.exe. We WANT to kill that cmd on Ctrl+C
#       (Phase 47 gap-closure-13 behavior — otherwise cmd shows
#       "Terminate batch job (Y/N)?" and waits for user input).
#   (b) Operator types ./start.bat in an existing PowerShell/cmd session
#       -> parent is the operator's WORKING shell (or a cmd transient that
#       ultimately reports back to that working shell). Killing it would
#       close the operator's session and lose unsaved work in their other
#       editor tabs / consoles.
#
# Detection: walk the topmost contiguous cmd ancestor and look at ITS
# parent. If that grandparent is explorer.exe -> 'fresh-cmd'. Anything
# else (powershell.exe, pwsh.exe, WindowsTerminal.exe, conhost.exe with no
# explorer above, walk truncates, CIM/WMI all fail) -> 'existing-shell'.
#
# Safety bias: default to 'existing-shell' on any uncertainty. False-negative
# cost = one batch-prompt on a wegwerf window (recoverable). False-positive
# cost = operator's working shell gets force-killed (data loss). Always
# lean safe.
$script:InvocationContext = 'existing-shell'
try {
  if ($script:AncestorCmdPids.Count -gt 0) {
    # Topmost cmd in our captured ancestor chain.
    $topCmdPid = $script:AncestorCmdPids[-1]
    $topCmdParent = Get-ParentProcessId -ChildPid $topCmdPid
    if ($topCmdParent -gt 0) {
      $topCmdParentProc = Get-Process -Id $topCmdParent -ErrorAction SilentlyContinue
      if ($topCmdParentProc -and $topCmdParentProc.ProcessName -eq 'explorer') {
        $script:InvocationContext = 'fresh-cmd'
      }
    }
  }
} catch {
  # Stay 'existing-shell' on any error — safer.
}
Write-Host "[start]    Invocation context: $script:InvocationContext (ancestor cmds: $($script:AncestorCmdPids.Count))"
```

| Get-CimInstance / Get-WmiObject -> ancestor process inspection | Reading process metadata from local WMI — local kernel data, no network surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-49-01 | Tampering | $script:CtrlHandlerDelegate GC reclamation | mitigate | Delegate instance stored in $script: scope keeps GC root alive for the lifetime of the PS process. If reclaimed, Windows would call freed memory — process crash, not security issue, but reliability. Verified by acceptance criterion grep count >= 2. |
| T-49-02 | Denial of Service | Operator's working shell killed by false-positive 'fresh-cmd' detection | mitigate | Default-safe: $script:InvocationContext defaults to 'existing-shell'. Detection only flips to 'fresh-cmd' on positive proof (topmost cmd's parent process name equals 'explorer'). Any walk/CIM error stays safe. |
| T-49-03 | Tampering | Get-ParentProcessId returning attacker-controlled PID | accept | Local-only inspection of WMI; an attacker who can MITM Win32_Process queries already has SYSTEM. Out of threat model for a click-and-run launcher on the operator's own machine. |
| T-49-04 | Elevation of Privilege | SetConsoleCtrlHandler P/Invoke requires admin? | accept | Documented Win32 API, no admin requirement — runs in the same access token as the calling process. Confirmed by Microsoft Learn docs. |
| T-49-05 | Information Disclosure | Invocation-context banner line leaks ancestor cmd count | accept | Numeric count of cmd ancestors is observable via Get-Process anyway; no secret leaked. Useful for operator triage. |
| T-49-06 | Denial of Service | Ctrl+C delegate returns $false -> Windows default handler kills us before cleanup runs | mitigate | The CancelKeyPress handler sets $e.Cancel = $true, blocking default action so $cleanup runs first. The SetConsoleCtrlHandler handler intentionally returns $false so the kernel KILL_ON_JOB_CLOSE belt fires after our cleanup — designed redundancy, not a bug. |
