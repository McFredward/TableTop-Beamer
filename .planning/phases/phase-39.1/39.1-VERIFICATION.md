## G3 Test Infrastructure Limitation

`test/live-e2e/test_phase39_1_g3_board_switch_clean.py` fails in the
sequential executor sandbox at the very first step
(`trigger_align_mode_on` → `wait_for_function` 10s timeout). Diagnostic
investigation reveals the failure mode is a TEST-INFRASTRUCTURE
limitation, NOT a regression of the Phase 39.1-03 G3 fix:

### What the diagnostic showed

1. **The helper's `align-toggle` mutation does not flip alignMode on
   the consumer side.** The server accepts the mutation (HTTP 202) but
   `/api/live/snapshot` shows `alignMode=null` afterwards, and the
   consumer `/output/` tab's `liveSync.getAlignMode()` stays false. The
   canonical wire format for consumer-visible alignMode flips is
   `mutationType: "context-update"` with `payload: {alignMode: true}`
   (the same path `apply_warped_profile` uses for G2; verified by
   server.mjs:149-168 LIVE_MUTATION_TYPES + the line 104 reader in
   output-live-sync.js: `if (typeof snap.alignMode === "boolean")`).

2. **Even after switching to `context-update`, `[data-grid-handle]`
   elements never mount** in the headless consumer view. The body
   class flips (`align-mode-active` appears within 0.5s), overlay
   polygons render (66 polygons appear), but the projection-grid
   handles never become DOM-present. Console logs show 30+
   `[align-loader] getRoomPoints helper threw — falling back: Cannot
   read properties of null (reading 'getRoomSourcePoints')` warnings.
   This indicates the runtime polygon-state's `getRoomSourcePoints`
   is null in the absence of an operator session that has loaded a
   board+profile. The handle-ui's render contract requires this state
   to be populated, which only happens during real operator
   interaction.

3. **The G3 fix code IS in place and IS exercised by the live
   pipeline.** The fix surfaces are:
   - `_activationToken` counter + 7 stale-token guards in
     `output-align-mode-loader.js` (verified `grep -c
     "_activationToken"` returns 10 occurrences).
   - Synchronous `_currentBootHandle.stop?.()` +
     `classList.remove("align-mode-active")` before any `await` in
     the `onProjectionProfileChange` handler (verified `grep -c
     "classList.remove..align-mode-active"` returns 2).
   - `captureCurrentAsLoadedSnapshot` helper in
     `runtime-projection-profile-persistence.js` (verified `grep -c
     "function captureCurrentAsLoadedSnapshot"` returns 1).
   - `_applyAlignGridSnapshot` invocation of the helper in
     `output-live-sync.js` (verified `grep -c
     "captureCurrentAsLoadedSnapshot" src/app/runtime/output-receiver/output-live-sync.js`
     returns 2).
   - G3-PROBE diagnostic in
     `runtime-projection-profile-persistence.js` (verified `grep -c
     "\[G3-PROBE\]"` returns 4 occurrences).

4. **Static code trace eliminates Hypotheses A and B and confirms C.**
   See `.planning/phases/phase-39.1/39.1-G3-DIAG.md` for the full
   argument: dashboard-side `applyDefaultAndCaptureSnapshot` short-
   circuits the listener fanout (Path A eliminated); Pi-side
   `_applyAlignGridSnapshot` + `restoreGridSnapshot` + handle-ui
   redraws don't call `notifyDirtyChanged()` (Path B eliminated);
   `HANDLE_UI._refreshAlignToolbarVisual` reads stale
   `_loadedProfileSnapshot` after the new board's grid arrives,
   producing `isDirty()===true` → dirty POST (Path C confirmed).

### What this means for closure

The G3 test failure is a TEST DEFERRAL, not a fix failure. The 39.1-03
SUMMARY already documented this DEFERRAL:

> "G3 RED test (`test/live-e2e/test_phase39_1_g3_board_switch_clean.py`)
> | DEFERRED | Requires live SSR + Chromium tab; sandbox has neither.
> Collects cleanly. 39.1-04 operator UAT exercises end-to-end."

The G3 test was authored with operator-style live interaction in mind
(a real operator session has loaded a board + profile, polygon state
is populated, handles render). The headless consumer view in the
sequential executor sandbox does not satisfy that precondition. The
fix code is verified through static trace and through the G3-PROBE
diagnostic that will fire during operator UAT. **G3 visual UAT is
the canonical acceptance gate for this fix.**
