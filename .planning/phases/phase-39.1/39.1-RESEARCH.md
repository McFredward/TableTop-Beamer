### Why Phase 39-4 was wrong-layer

Phase 39-4 added the UV clamp at the FRAGMENT-SHADER sampling stage. Its purpose is to keep the LINEAR-filter's 2×2 texel tap-window from straddling the outer edge of the source texture and reading the CLAMP_TO_EDGE wrap value at one tap. That's a real fix for the EDGES of the texture, but **inter-cell seams are NOT a sampling problem**:

- All cells sample the SAME continuous source texture (the page render).
- The cell-boundary discontinuity originates from the RASTERIZER deciding which triangle owns each pixel at the shared edge, per the diamond-exit rule (per Wikipedia/OpenGL.org rasterization spec: a pixel is rasterized by a primitive only if the line/edge exits the diamond inscribed within the pixel — and adjacent triangles see the edge in OPPOSITE directions, so a pixel right on the shared edge can be claimed by BOTH (overdraw) or NEITHER (gap), depending on sub-pixel exact coordinates).
- Phase 30 pixel-snap (lines 485-499) ensures vertices land on integer pixel coords so the shared edge is EXACTLY at an integer pixel boundary, but that doesn't resolve which triangle owns THAT pixel — fill-rule arbitrariness remains. Both triangles' interpolated UVs differ by sub-texel amounts at that pixel, the LINEAR-filter samples slightly different colors from the source canvas, and the result is a 1-px discontinuity that JPEG (q=60) smooths into invisible but H.264 macroblock-quantizes back into a visible ridge.

[CITED: https://www.opengl.org/archives/resources/faq/technical/rasterization.htm — diamond-exit rule. CONFIRMED.]

[CITED: https://community.khronos.org/t/issues-with-triangle-seams/106969 — practical OpenGL community discussion of triangle-shared-edge seams; standard mitigation is geometry inflate (overlap) or seamless texture coordinates.]

The Phase 39-1 verification test used `Page.captureScreenshot` (JPEG q=60) — a JPEG decoder using 8×8 DCT blocks with its OWN quantization characteristics that PRESERVE single-pixel ridges less than H.264's 4×4 transform blocks do. The test legitimately saw `max_delta=0` because the per-line mean-aggregated delta (Phase 39-4 test repair commit d081898) is below the JPEG-baseline noise. On the H.264 path the macroblock-boundary quantization snaps single-pixel residual into visible structured noise across a full block. Same image, different artifact.

#### Sub-G2b: 2-pixel mesh-cell-overlap (vertex inflate)

**Goal:** ensure each pixel along a shared cell edge is COVERED BY BOTH adjacent triangles (overdraw), eliminating diamond-exit ambiguity. Since both triangles render the SAME continuous source canvas (no per-cell coloring), the overdraw is invisible — the post-overlap pixel value is identical regardless of which triangle "wins".

**Algorithm (the load-bearing detail):**

Each grid cell is a quadrilateral split into two triangles (TL-TR-BR and TL-BR-BL — see `runtime-projection-gl-renderer.js:447-453`). Each cell's two triangles share the diagonal TL-BR. EACH cell shares its top edge with the cell above, its bottom edge with the cell below, its left edge with the cell to the left, its right edge with the cell to the right.

**Approach: per-triangle dilate by 1 framebuffer-pixel in the outward normal direction.** Modify `runtime-projection-gl-renderer.js`'s position-buffer build (lines 485-507) to compute, FOR EACH TRIANGLE (not per-vertex — same vertex appears in 6 triangles in interior grid cells, and each triangle wants a DIFFERENT dilate direction), three vertex positions where each vertex is shifted outward perpendicular to the OPPOSITE edge.

**The implementation:** instead of indexed-draw with `gl.drawElements`, switch to flat `gl.drawArrays(gl.TRIANGLES, 0, triCount*3)`. Each triangle gets 3 unique vertices. For each triangle, compute the centroid, then for each vertex compute the unit vector from centroid to vertex and shift the vertex outward by `1.0` framebuffer pixel along that vector (or equivalently, by `(2.0/bufW, 2.0/bufH)` in NDC space per axis — but the per-axis NDC shift doesn't preserve direction; better to compute in pixel space then map back to NDC).

**Concrete pseudo-code for the vertex-build loop:**

```js
// Replace the indexed-draw vertex build with per-triangle dilated vertices.
const trisOut = []; // 3 vertices * 2 floats * triCount
const uvsOut = [];

for (let row = 0; row < rows - 1; row++) {
  for (let col = 0; col < cols - 1; col++) {
    const tl = { px: Math.round(getPoint(row, col).x * bufW),     py: Math.round(getPoint(row, col).y * bufH) };
    const tr = { px: Math.round(getPoint(row, col + 1).x * bufW), py: Math.round(getPoint(row, col + 1).y * bufH) };
    const bl = { px: Math.round(getPoint(row + 1, col).x * bufW), py: Math.round(getPoint(row + 1, col).y * bufH) };
    const br = { px: Math.round(getPoint(row + 1, col + 1).x * bufW), py: Math.round(getPoint(row + 1, col + 1).y * bufH) };
    // UV is grid.srcXs[col], 1 - grid.srcYs[row] (with UNPACK_FLIP_Y_WEBGL).
    const uvTL = { u: grid.srcXs[col],     v: 1 - grid.srcYs[row] };
    const uvTR = { u: grid.srcXs[col + 1], v: 1 - grid.srcYs[row] };
    const uvBL = { u: grid.srcXs[col],     v: 1 - grid.srcYs[row + 1] };
    const uvBR = { u: grid.srcXs[col + 1], v: 1 - grid.srcYs[row + 1] };
    
    // Triangle 1: TL, TR, BR — dilate by 1 px from centroid.
    pushDilatedTri(trisOut, uvsOut, tl, tr, br, uvTL, uvTR, uvBR, bufW, bufH, 1.0);
    // Triangle 2: TL, BR, BL — dilate by 1 px from centroid.
    pushDilatedTri(trisOut, uvsOut, tl, br, bl, uvTL, uvBR, uvBL, bufW, bufH, 1.0);
  }
}

function pushDilatedTri(trisOut, uvsOut, a, b, c, uvA, uvB, uvC, bufW, bufH, dilatePx) {
  const cx = (a.px + b.px + c.px) / 3;
  const cy = (a.py + b.py + c.py) / 3;
  for (const [pt, uv] of [[a, uvA], [b, uvB], [c, uvC]]) {
    const dx = pt.px - cx;
    const dy = pt.py - cy;
    const len = Math.hypot(dx, dy) || 1;
    const sx = pt.px + (dx / len) * dilatePx;
    const sy = pt.py + (dy / len) * dilatePx;
    // Map to NDC; UNPACK_FLIP_Y_WEBGL flips Y so y maps as `1 - py/bufH * 2`.
    trisOut.push((sx / bufW) * 2 - 1, 1 - (sy / bufH) * 2);
    // UV stays the same — we're stretching the triangle slightly in position
    // space, NOT in UV space. The fragment that lands at the "extra" outer
    // pixel will sample a UV that's an EXTRAPOLATION of the triangle's
    // interpolated UV. Since adjacent cells share the boundary UV exactly,
    // the EXTRAPOLATED UV at the seam is identical from both sides ⇒ same
    // source-canvas color sampled ⇒ no visible seam discontinuity. (This
    // is the load-bearing claim: it works because UVs are continuous
    // across the cell boundary, not because positions are.)
    uvsOut.push(uv.u, uv.v);
  }
}
```

**Why this works:** at the shared edge between two cells, the "outer" pixel of cell-A (the boundary pixel) is also the "outer" pixel of cell-B. Both triangles now contain that pixel. Both interpolate UV from their cell-corners — and since cell-A's right-edge UV equals cell-B's left-edge UV (the grid is consistent), the SAMPLED UV at the boundary pixel is identical from both triangles. Both triangles produce the same fragment color. Overdraw is a no-op.

**Caveat: dst-alpha=0 leak.** If the GL framebuffer cleared with `clearColor(0,0,0,0)` (dashboard path, line 546-547), and a dilated triangle pushes a vertex OUTSIDE the canvas rect, then the dilated pixel might be at NDC coordinate beyond [-1, 1]. WebGL clips it cleanly (it's outside the viewport). On `/output/` (`clearColor(0,0,0,1)`, line 544) there's no alpha issue.

**Caveat: outer-boundary dilation.** Cells on the OUTSIDE edges of the grid (row=0, row=rows-1, col=0, col=cols-1) will dilate their outer vertex OUTWARD of the canvas. This is harmless — the dilated portion clips against the viewport and is discarded. The "visible" boundary remains at the original (un-dilated) outer-edge vertex of the next cell inward, which is unchanged.

**Caveat: triangle-diagonal seam (TL-BR).** Within a cell, the diagonal between the two triangles (TL-BR) is ALSO a shared edge. After dilation, both triangles dilate AWAY from each other along that diagonal — same overdraw mechanism applies. Verified: any sub-pixel diamond-exit ambiguity along the diagonal is now resolved by both triangles claiming the pixel.

**Phase 39-4 UV-inset interaction.** The shader currently does `clamp(vUV, 0.5/uTexSize, 1.0 - 0.5/uTexSize)` (line 294). For dilated triangles, the EXTRAPOLATED UV at the dilated pixel may exceed `[0, 1]` (specifically, on edge-of-grid cells, the dilated outer vertex extrapolates UV slightly outside [0,1]). The clamp catches this and snaps to the inset boundary. NO INTERACTION ISSUE — clamp simply does its job. The Phase 39-4 fix continues to defend the texture's outer boundary, AND the new dilate defends inter-cell boundaries. Composable.

**Phase 30 pixel-snap interaction.** Pixel-snap rounds vertex positions to integer framebuffer pixels BEFORE dilate. The dilate then SHIFTS by exactly +1 framebuffer pixel along the centroid-to-vertex direction. After dilate, the vertex is at a non-integer position (snap+dilate-unit-vector), but that's fine — the rasterizer's diamond-exit on a triangle whose vertices are slightly outside the pixel-grid is well-defined.

**Cached arrays + W12 invalidate path.** Phase 38 W12 invalidates `_glPositions / _glUVs / _glIndices` on `restoreGridSnapshot` (`runtime-projection-grid-state.js:266-268`). The new flat-arrays path replaces `_glIndices` with a per-triangle position-stream — the W12 invalidate must clear ALL THREE cached arrays. Carry-forward intact.

### Root cause #1 — handles render with fresh grid, overlay-lines absent

**The race:** the `activate()` triggered by `projectionProfileChange` is fully async — it awaits 4 things (loadBundle, refreshBoardCatalog, refreshSelectedBoard, import boot-handle-ui). Meanwhile, the dashboard's `autoLoadRememberedProjectionProfile` runs `void` (fire-and-forget). The receive path at `output-live-sync.js:209-256` calls:

**Critical:** if align-mode was ON before the board switch, the bundle and HANDLE_UI ARE already loaded. So the receive path actually mutates the grid AND redraws handles BEFORE the new `activate()` has set up the new `_state` and re-bound to the new boardId.

## Architecture Patterns

### Pattern 1: Per-triangle dilation for seam erasure
**What:** Each triangle in the mesh independently dilates its vertices outward by 1 framebuffer-pixel along the centroid-to-vertex unit vector. Adjacent triangles' dilated regions overlap. Shared-edge pixels are covered by BOTH triangles; overdraw with identical interpolated UV produces no visible discontinuity.
**When to use:** Any mesh-rendered surface where rasterizer diamond-exit ambiguity at shared edges produces visible artifacts after a lossy compression step.
**Example:** see §G2 sub-G2b code block above.

### Pattern 2: Synchronous teardown + activation token for async re-mount
**What:** Async re-mount sequences MUST tear down the prior state synchronously BEFORE any await. Track an activation token; stale activations exit early on token mismatch.
**When to use:** Any module that exposes both an `activate()` async function AND can be re-invoked while a prior activation is in-flight (board switch, profile change, viewport reconfiguration).
**Example:** see §G3 sub-G3a.

### Pattern 3: Authority-pushed-snapshot is the new baseline
**What:** When a consumer module receives a snapshot from the authoritative source, it should re-baseline its local "dirty-against-snapshot" state immediately. Otherwise, the local dirty flag becomes stale relative to the actual server state.
**When to use:** Any read-only consumer that tracks a "loaded baseline" for dirty-flag purposes.
**Example:** see §G3 sub-G3b `captureCurrentAsLoadedSnapshot`.

### Anti-Patterns to Avoid

- **Verifying H.264 stream output via JPEG-CDP screenshot.** Different compression pipelines, different artifacts. Phase 39-1 verified this is an explicit anti-pattern. (CRITICAL — operator-flagged.)
- **Layering a shader fix (UV inset) for a rasterizer-side problem (diamond-exit).** Phase 39-4 verified this is an explicit anti-pattern. UV inset is a real fix for outer-boundary sampling, NOT for inter-cell rasterizer ambiguity. (CRITICAL — operator-flagged.)
- **Fire-and-forget `await activate()` inside `onProjectionProfileChange` without first stopping prior state.** Phase 39 verification caught this; Phase 39.1 fixes it.
- **Bumping `serverRendering.qualityPreset` and assuming the publisher's `maxBitrate` changes.** The publisher derives bitrate independently from `effectiveStreamFpsCap`. Without explicit plumbing of `encoderConfig.bitrate` into `deriveSimulcastBitrates`, the config change is cosmetic.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| H.264 deblocking on raw stream | Custom DCT-residual smoother in shader | Raise encoder bitrate; let H.264's built-in in-loop deblocking filter do its job. | Project doesn't expose ffmpeg / x264 encoder internals; we control bitrate (input) not deblocking-filter strength (encoder param). |
| Mesh seam erasure via texture-atlas seam-aware tessellation | Liu et al. SIGGRAPH 2017 "Seamless" paper | Per-triangle dilation (this phase) | We don't have a UV-atlas; we have a continuous-source-canvas mesh-warp. The simpler dilation works because UVs are continuous across cell boundaries. |
| WebRTC ICE re-negotiation on bitrate change | Hand-rolled SDP munging | `RTCRtpSender.setParameters({ encodings: [...] })` on the existing producer | Standard WebRTC API; mediasoup supports it natively. (NOT needed in this phase — bitrate is set at producer creation time.) |
| Async cancellation | Custom flag-checked-everywhere | Activation-token counter idiom (project convention) OR JS standard AbortController | Project precedent uses counter; consistent style trumps marginal API ergonomics. |

**Key insight:** the operator's "real" pipeline (H.264 WebRTC → Pi decoder → projector) is qualitatively different from any dev-box test pipeline (JPEG screenshot, raw canvas read, headless render). Every G2 test must touch the actual decode path on the consumer's hardware (or a faithful simulation thereof via `<video>` element + Playwright).

```js
// In runtime-projection-gl-renderer.js, replace the indexed-draw vertex build
// (current lines 426-456) with this flat per-triangle dilated build.
// Source: this RESEARCH §G2 sub-G2b.

const triCount = (rows - 1) * (cols - 1) * 2;
const verticesPerTri = 3;
const floatsPerVertex = 2;

if (!_glPositions || _glPositions.length !== triCount * verticesPerTri * floatsPerVertex
    || _glCachedRows !== rows || _glCachedCols !== cols) {
  _glPositions = new Float32Array(triCount * verticesPerTri * floatsPerVertex);
  _glUVs = new Float32Array(triCount * verticesPerTri * floatsPerVertex);
  _glIndices = null;  // unused in flat-draw mode
  _glCachedRows = rows;
  _glCachedCols = cols;
}

const DILATE_PX = 1.0;  // operator-spec'd "1-2 extra pixels"; tunable.
let vi = 0;

function pushVertex(pt_px, pt_py, cx, cy, u, v) {
  const dx = pt_px - cx;
  const dy = pt_py - cy;
  const len = Math.hypot(dx, dy) || 1;
  const sx = pt_px + (dx / len) * DILATE_PX;
  const sy = pt_py + (dy / len) * DILATE_PX;
  _glPositions[vi]     = (sx / bufW) * 2 - 1;
  _glPositions[vi + 1] = 1 - (sy / bufH) * 2;
  _glUVs[vi]     = u;
  _glUVs[vi + 1] = v;
  vi += 2;
}

for (let row = 0; row < rows - 1; row++) {
  for (let col = 0; col < cols - 1; col++) {
    const tl_pt = getPoint(row, col),     tr_pt = getPoint(row, col + 1);
    const bl_pt = getPoint(row + 1, col), br_pt = getPoint(row + 1, col + 1);
    const tl_px = Math.round(tl_pt.x * bufW), tl_py = Math.round(tl_pt.y * bufH);
    const tr_px = Math.round(tr_pt.x * bufW), tr_py = Math.round(tr_pt.y * bufH);
    const bl_px = Math.round(bl_pt.x * bufW), bl_py = Math.round(bl_pt.y * bufH);
    const br_px = Math.round(br_pt.x * bufW), br_py = Math.round(br_pt.y * bufH);
    const u_l = grid.srcXs[col],     u_r = grid.srcXs[col + 1];
    const v_t = 1 - grid.srcYs[row], v_b = 1 - grid.srcYs[row + 1];
    
    // Triangle 1: TL, TR, BR
    {
      const cx = (tl_px + tr_px + br_px) / 3;
      const cy = (tl_py + tr_py + br_py) / 3;
      pushVertex(tl_px, tl_py, cx, cy, u_l, v_t);
      pushVertex(tr_px, tr_py, cx, cy, u_r, v_t);
      pushVertex(br_px, br_py, cx, cy, u_r, v_b);
    }
    // Triangle 2: TL, BR, BL
    {
      const cx = (tl_px + br_px + bl_px) / 3;
      const cy = (tl_py + br_py + bl_py) / 3;
      pushVertex(tl_px, tl_py, cx, cy, u_l, v_t);
      pushVertex(br_px, br_py, cx, cy, u_r, v_b);
      pushVertex(bl_px, bl_py, cx, cy, u_l, v_b);
    }
  }
}

// Bind + draw — note drawArrays not drawElements.
_gl.bindBuffer(_gl.ARRAY_BUFFER, _glPosBuf);
_gl.bufferData(_gl.ARRAY_BUFFER, _glPositions, _gl.DYNAMIC_DRAW);
_gl.enableVertexAttribArray(_glAttrPos);
_gl.vertexAttribPointer(_glAttrPos, 2, _gl.FLOAT, false, 0, 0);

_gl.bindBuffer(_gl.ARRAY_BUFFER, _glUVBuf);
_gl.bufferData(_gl.ARRAY_BUFFER, _glUVs, _gl.DYNAMIC_DRAW);
_gl.enableVertexAttribArray(_glAttrUV);
_gl.vertexAttribPointer(_glAttrUV, 2, _gl.FLOAT, false, 0, 0);

// drawArrays — no element-buffer needed.
_gl.drawArrays(_gl.TRIANGLES, 0, triCount * verticesPerTri);
```

### Bitrate plumbing (G2 sub-G2a)

```js
// Source: this RESEARCH §G2 sub-G2a. Edit src/server/ssr-stream-publisher.mjs.

function deriveSimulcastBitrates({ effectiveStreamFpsCap = 60, encoderConfig = null } = {}) {
  let total;
  const envOverride = Number(process.env.SSR_INITIAL_BITRATE);
  const configOverride = Number(encoderConfig?.bitrate);
  // Phase 39.1 G2a: take max of (env, config, fps-derived) so a preset bump
  // in config can ONLY raise bitrate, never lower it below the fps-derived
  // default.
  let fpsDerived;
  if (effectiveStreamFpsCap >= 60)      fpsDerived = 16_000_000;
  else if (effectiveStreamFpsCap >= 45) fpsDerived = 12_000_000;
  else                                  fpsDerived = 8_000_000;
  if (Number.isFinite(envOverride) && envOverride > 0) {
    total = envOverride;
  } else {
    total = Math.max(
      Number.isFinite(configOverride) && configOverride > 0 ? configOverride : 0,
      fpsDerived,
    );
  }
  return {
    low:  Math.round(total * 0.15),
    mid:  Math.round(total * 0.35),
    high: total,
    total,
  };
}
```
