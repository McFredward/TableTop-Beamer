# Phase 39.1 Plan 3: Wave 1 — G3 Fix (Sync Teardown + Activation Token + captureCurrentAsLoadedSnapshot) Summary

One-liner: Pi /output/'s board-switch race is closed via two parallel
fixes — `onProjectionProfileChange` now tears down `_currentBootHandle`
synchronously before any await + removes the `align-mode-active` body
class so HANDLE_UI hides during the async re-mount window (closes
Hypothesis-1 "handles without overlay"), AND `_applyAlignGridSnapshot`
re-baselines persistence's `_loadedProfileSnapshot` via the new
`captureCurrentAsLoadedSnapshot` helper so HANDLE_UI's
`_refreshAlignToolbarVisual` reads `isDirty()===false` immediately
after a server-pushed grid arrives (closes Hypothesis-C "spurious
dirty flag"). Defense-in-depth `_activationToken` counter aborts
stale in-flight `activate()` invocations across all 6 await points.

## What Was Built

### Sub-G3a — synchronous teardown + activation token (1 file)

`src/app/runtime/output-receiver/output-align-mode-loader.js` (56 LOC
added):

1. Module-level `let _activationToken = 0;` declared alongside
   `_currentBootHandle`. Counter is monotonic across the module's
   lifetime; JS Number range (2^53) supports millennia of board
   switches at any realistic operator rate (T-39.1-03-02 disposition
   `accept`).
2. `activate()` captures `const myToken = ++_activationToken;` right
   after the `stopped` guard at entry. Each `await` in the body is
   immediately followed by `if (myToken !== _activationToken) return;`
   — at:
   - `await loadBundleOnce()` (line ~476)
   - `await refreshBoardCatalog()` conditional (line ~478)
   - `await refreshSelectedBoard()` conditional (line ~480)
   - `await import("/src/app/runtime/output-receiver/boot-handle-ui.js")`
     (line ~544 — the widest await window at 50-500ms cold)
   - `await fetch("/api/live/snapshot")` inside W4 drain
   - `await resp.json()` inside W4 drain
3. `onProjectionProfileChange` handler (formerly lines 807-816) now
   opens with synchronous teardown BEFORE any await:
   ```js
   if (_currentBootHandle) {
     try { _currentBootHandle.stop?.(); } catch { /* listener guard */ }
     _currentBootHandle = null;
   }
   document.body.classList.remove("align-mode-active");
   await refreshBoardCatalog();
   // ... rest unchanged
   ```
   The `_currentBootHandle.stop()` call reuses Phase 38 W11's
   ordered-teardown path (HANDLE_UI.onAlignModeChange(false) BEFORE
   unsubscribe), so this synchronous invocation is identical in
   semantics to the pre-existing path — only the timing moves from
   "after async re-mount" to "before async re-mount".

### Sub-G3b — captureCurrentAsLoadedSnapshot helper + wire (2 files)

`src/app/runtime/viewport/runtime-projection-profile-persistence.js`
(34 LOC added):

1. New helper after `applyDefaultAndCaptureSnapshot` (line ~728):
   ```js
   function captureCurrentAsLoadedSnapshot(name) {
     if (!_gridStateApi) return;
     _loadedProfileName = name || null;
     _loadedProfileSnapshot = _gridStateApi.snapshotGridState();
     try { _persistLoadedProfileToLs?.(); } catch (_e) { /* LS quota */ }
     _recomputeAndNotifyDirty();
   }
   ```
2. Public exports object (the `window.TT_BEAMER_RUNTIME_PROJECTION_PROFILE_PERSISTENCE`
   assignment around line 800) gains `captureCurrentAsLoadedSnapshot`
   grouped with the snapshot-related helpers.

`src/app/runtime/output-receiver/output-live-sync.js` (22 LOC added):

In `_applyAlignGridSnapshot` (~line 220), immediately after the
`gs.restoreGridSnapshot({...})` call AND after the
`if (Number.isFinite(snapAt)) _lastAppliedSnapAtMs = snapAt;` line
AND before the `ui.rebuildHandleElements?.()` call:

```js
try {
  const persist = window.TT_BEAMER_RUNTIME_PROJECTION_PROFILE_PERSISTENCE;
  if (persist?.captureCurrentAsLoadedSnapshot) {
    persist.captureCurrentAsLoadedSnapshot(
      snap?.profileId ?? snap?.name ?? null,
    );
  }
} catch (_e) { /* persistence module not loaded yet — safe to skip */ }
```

The lookup-at-call-time pattern handles the load-order race where
output-live-sync's WS dispatch path is wired BEFORE the lazy IIFE
bundle (which loads persistence.js) finishes parsing. During the
race window the call is a no-op; the next remote snapshot or the
W4 post-bundle drain catches up.

## Hypothesis Decision (Task 1)

**Decision:** Hypothesis C ([ASSUMED] confirmed by static code trace;
live operator reproduction deferred to 39.1-04 UAT per the plan's
`<probe_first>` directive).

**Why C, not A:** Dashboard's `applyDefaultAndCaptureSnapshot` captures
`_loadedProfileSnapshot = _gridStateApi.snapshotGridState()` BEFORE
calling `_recomputeAndNotifyDirty()`, so `isDirty()` returns false
(snapshot just captured == current grid). `_dirty` stays false. The
listener-fanout at lines 219-226 is short-circuited by the
`if (next === _dirty) return` guard at line 221 → no listener fires →
no dirty POST. **Path A eliminated.**

**Why C, not B:** `output-live-sync.js`'s `_applyAlignGridSnapshot`
calls `gs.restoreGridSnapshot(...)` then handle-ui's redraws.
`restoreGridSnapshot` and the four UI redraws don't call
`notifyDirtyChanged()` (verified by grep — only drag-event handlers
do). The W4 drain path takes the same route. **Path B eliminated.**

**Why C:** When boot-handle-ui activates (output-align-mode-loader.js:
~544), its `_offAlignModeChange = liveSync.onAlignModeChange(...)`
subscription fires `HANDLE_UI.onAlignModeChange(true)` → `showHandles()`
→ `_refreshAlignToolbarVisual()` → reads `persistence.isDirty()`.
`isDirty()` (lines 205-217) compares `_loadedProfileSnapshot` (OLD
board's, never re-baselined) against `_gridStateApi.snapshotGridState()`
(NEW grid post-apply). They differ → `isDirty()` returns true → the
dirty-listener fanout posts dirty=true to the dashboard.

39.1-G3-DIAG.md contains a decision tree mapping the actual G3-PROBE
stack capture (39.1-04 operator UAT) to the correct fix branch. If
Hypothesis A or B is captured instead, the captureCurrentAsLoadedSnapshot
helper is a harmless re-baseline that doesn't introduce regression
but doesn't fix the operator's symptom — at which point 39.1-04
re-routes to the alternative fix.
