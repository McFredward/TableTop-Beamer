# Phase 39.1 Plan 2: Wave 1 — G2 Combined Fix (Bitrate + Vertex Dilation)

One-liner: H.264 stream now runs at 24 Mbps via a new `ultra-high`
QUALITY_PRESETS entry plumbed through `deriveSimulcastBitrates` with
Math.max-of-three semantics; the mesh-warp renderer's draw call
switched from indexed `gl.drawElements` to flat per-triangle
`gl.drawArrays` with each vertex dilated 1 framebuffer-pixel outward
along the centroid-to-vertex unit vector — adjacent triangles'
dilated regions overlap at their shared edge with identical UVs,
eliminating the rasterizer's diamond-exit ambiguity that was
producing operator-visible seams in Phase 39.

## What Was Built

### Sub-G2a: H.264 bitrate bump (3 files)

`src/server/ssr-render-host.mjs` — `QUALITY_PRESETS` gains:

```js
"ultra-high":   { bitrate: 24_000_000, fpsTarget: 30, keyframeIntervalSec: 2, x264Preset: "fast" },
```

`src/server/ssr-stream-publisher.mjs` — `deriveSimulcastBitrates`
gains an `encoderConfig` parameter and uses
`Math.max(env, configOverride, fpsDerived)` semantics. The env var
`SSR_INITIAL_BITRATE` still wins when set; otherwise the config preset
can RAISE bitrate above the fps-derived default (16 Mbps at 60fps)
but never lower it. Pre-Phase-39.1, the qualityPreset bitrate field
was logging-only — never consumed by the actual RTP encoder cap.

`buildInPagePublisherScript` now passes `encoderConfig` to
`deriveSimulcastBitrates`. Single-layer maxBitrate at the produce
call (`bitrates.high`) automatically picks up the new total.

`config/global-defaults.json` — `serverRendering.qualityPreset` bumped
from `"balanced"` (8 Mbps) to `"ultra-high"` (24 Mbps).

**Net effect:** H.264 stream maxBitrate rises from 16 Mbps (60fps
fps-derived default) to 24 Mbps. Pi 4 hardware decoder rated for
H.264 level 4.2 (>50 Mbps) — 1.5x headroom over the operator's
previously rock-stable 16 Mbps per Plan §<action>5 + 39.1-RESEARCH
§G2 R2.4.

### Sub-G2b: Per-triangle vertex dilation (1 file)

`src/app/runtime/viewport/runtime-projection-gl-renderer.js` —

1. New file-level constant `const DILATE_PX = 1.0;` near the GL state
   declarations.
2. Vertex+index allocation block now allocates `triCount * 3 * 2` floats
   for BOTH positions AND UVs (flat-arrays per-triangle stream); index
   build removed. `_glIndices = null` in this mode; `_glIndexCount = 0`.
3. Per-vertex Phase-30 pixel-snap loop replaced with per-CELL emit. For
   each cell, two triangles share the TL-BR diagonal. Each triangle's
   centroid is the mean of its three pixel-snapped corners; each vertex
   is shifted outward by DILATE_PX framebuffer-pixels along the
   centroid->vertex unit vector. UVs remain at the un-dilated
   cell-corner values, including the UNPACK_FLIP_Y_WEBGL convention
   (`v = 1 - grid.srcYs[row]`).
4. Draw call: `_gl.drawElements(...)` -> `_gl.drawArrays(_gl.TRIANGLES, 0, triCount * 3)`.
5. Removed the `_gl.bindBuffer(_gl.ELEMENT_ARRAY_BUFFER, _glIdxBuf)`
   bind preceding the draw call (unnecessary in drawArrays mode).
   `_glIdxBuf` itself remains allocated.

**Preserved unchanged:**

- **Phase 30 B1 pixel-snap (`Math.round`):** runs as the INPUT to
  centroid computation and dilation. Snapping ensures stable centroid
  math under grid drag.
- **Phase 39-4 D-03 sub-path B UV-inset shader (`clamp(vUV, 0.5/uTexSize,
  ...)`)**: completely untouched. Defends the texture outer boundary;
  the dilation operates in NDC position space only and is orthogonal
  to the UV space the shader clamps. d03 carry-forward test (3x3,
  5x5, 9x9 grids) all GREEN post-edit — confirms the UV-inset shader
  still works exactly as it did in Phase 39-4.
- **Phase 38 W12 `invalidateCachedArrays`:** the renderer-side function
  (lines 606-613 of the modified file) already nulls `_glPositions`
  and `_glUVs`. The call site in `grid-state.js` `restoreGridSnapshot`
  is unchanged. `_glIndices` is now permanently null; the existing
  invalidate function tolerates this. W12 carry-forward test GREEN.

## invalidateCachedArrays — was an edit needed?

**No.** The plan §<action>7 asked the executor to verify whether
`invalidateCachedArrays` already nulls `_glPositions` and `_glUVs`,
and edit it if not. Verified by reading lines 606-613 of the
pre-edit file:

```js
function invalidateCachedArrays() {
  _glCachedRows = 0;
  _glCachedCols = 0;
  _glPositions = null;
  _glUVs = null;
  _glIndices = null;
  _glIndexCount = 0;
}
```

All three array fields are nulled, plus the row/col caches. The
function tolerates `_glIndices` being permanently null in the new
flat-draw mode (it's overwritten with null on next call regardless).
**No edit to `invalidateCachedArrays` was required.**

## DILATE_PX value at commit

**`DILATE_PX = 1.0`** (the Plan §<action>2 baseline). Operator-spec
allowed 1-2 pixels; the Plan recommended starting at 1.0 because:

- 1 framebuffer-pixel is the smallest non-zero dilation that can
  guarantee both triangles cover the boundary pixel under any
  rasterizer coverage rule.
- Larger values widen the overdraw band, which is wasted GPU work
  and (at extreme grids) could push UV interpolation further from
  the geometric center, potentially making the UV-inset shader's
  outer-boundary clamp do more work.

Plan §<acceptance_criteria> permits bumping to 1.5 or 2.0 in 39.1-04
if the operator UAT reports residual interior seams. Wave 1's G2
RED test (test_phase39_1_g2_h264_no_seams.py) passed at
DILATE_PX=1.0 in 79 seconds wall (single iteration, no retry), so
no bump is necessary for the automated gate.
