### G2 — Mesh-Warp Seams STILL Visible in Operator's H.264 Stream

**Operator-Quote (Phase 39 UAT 2026-05-13):**
> "Die Streifen sind weiterhin da und sichtbar bei Animationen,
> insbesondere solid-color."

**Root cause analysis (39.1-RESEARCH §G2):**

Phase 39-4's UV-inset shader fix
(`clamp(vUV, 0.5/uTexSize, 1-0.5/uTexSize)`) addresses only the
texture's OUTER boundary, NOT the inter-cell seams between mesh
cells. Phase 39's verification test
(`test/live-e2e/test_phase39_d03_no_seams.py`) captured JPEG-CDP
screenshots (quality=60), which use an 8×8 DCT block size that
smooths single-pixel ridges into invisibility — but the operator's
actual pipeline is `getDisplayMedia` → H.264 baseline → WebRTC → Pi
decode, which uses 4×4 transform blocks aligned to mesh-cell edges
and quantizes the same single-pixel residual into visible structured
noise. Same source image, different compression artifact pattern,
different visibility threshold.

**Fix applied (Plan 39.1-02 Wave 1):**

Combined two-part fix:

1. **H.264 bitrate bump to ultra-high preset (24 Mbps single-layer).**
   - `src/server/ssr-render-host.mjs` — new
     `QUALITY_PRESETS["ultra-high"] = { bitrate: 24_000_000,
     fpsTarget: 30, keyframeIntervalSec: 2, x264Preset: "fast" }`
   - `src/server/ssr-stream-publisher.mjs` —
     `deriveSimulcastBitrates({ effectiveStreamFpsCap, encoderConfig })`
     gains `Math.max(env, configOverride, fpsDerived)` semantics. Env
     var `SSR_INITIAL_BITRATE` still wins; otherwise the preset can
     RAISE bitrate above the fps-derived default but never lower it.
   - `config/global-defaults.json#serverRendering.qualityPreset` bumped
     from `"balanced"` (8 Mbps) to `"ultra-high"` (24 Mbps).
   - Commit `cc8823c`.

2. **Per-triangle 1-pixel vertex dilation in mesh-warp drawArrays.**
   - `src/app/runtime/viewport/runtime-projection-gl-renderer.js` —
     new `const DILATE_PX = 1.0` constant; switched from indexed
     `drawElements` to flat per-triangle `drawArrays` (vertex+UV
     stream allocates `triCount * 3 * 2` floats for both). Each
     triangle's three vertices are pixel-snapped (Phase 30 preserved)
     and then shifted outward along the centroid-to-vertex unit
     vector by DILATE_PX framebuffer-pixels. Adjacent triangles'
     dilated regions overlap at their shared edge with identical UVs
     — invisible overdraw eliminates rasterizer diamond-exit ambiguity.
   - Commit `e3ddd14`.

3. **Phase 39-4 UV-inset shader preserved** as the orthogonal
   texture-outer-boundary defense (verified: 3/3 parametric grids
   3×3 / 5×5 / 9×9 still pass `test_phase39_d03_no_seams.py` with
   max_delta=0).

**Verification evidence:**
- NEW G2 H.264 RED test `test/live-e2e/test_phase39_1_g2_h264_no_seams.py`:
  **RED → GREEN.** Single-iteration 70.3s pass. Captures pixels from

### G3 — Board-Switch Shows Partial Align Mode + Spurious Dirty Flag

**Operator-Quote (Phase 39 UAT 2026-05-13):**
> "Bei einem Wechsel vom Board wird, warum auch immer, Teile des
> Align Modes angezeigt (nur die Handles, nicht die Linien des
> Overlays) und ohne das ich was verändert hab wird die dirty flag
> getriggert was durch 'unsaved on /output/' dann verhindert, dass
> ich den Align Mode aktivieren kann."

**Root cause analysis (39.1-RESEARCH §G3 + 39.1-G3-DIAG.md):**

Two coupled races during board-switch:

1. **Handles-without-overlay race (Root cause #1, HIGH confidence):**
   `output-align-mode-loader.js`'s `onProjectionProfileChange`
   handler called `await activate()` async. `activate()` performed a
   dynamic `import("boot-handle-ui.js")` (50-500ms cold). During
   that window, the dashboard's `applyDefaultAndCaptureSnapshot`
   broadcast a `align-grid-snapshot`, which mutated the live grid
   AND ran the polygon-editor's renderRoomOverlay with the OLD
   board's polygons → handles visible, overlay-lines absent.

2. **Spurious dirty flag (Root cause #2 = Hypothesis C confirmed by
   code trace):** When the new bootHandleUi activates, its subscription
   to `onAlignModeChange` triggers
   `HANDLE_UI._refreshAlignToolbarVisual()`, which reads
   `persistence.isDirty()`. `isDirty()` compares
   `_loadedProfileSnapshot` (the OLD board's snapshot, never
   re-baselined) against `_gridStateApi.snapshotGridState()` (the NEW
   grid post-apply). They differ → `isDirty()` returns TRUE → the
   dirty-listener fanout posts dirty=true to the dashboard. This is
   why the dashboard shows "unsaved on /output/" without the operator
   making any change.

**Fix applied (Plan 39.1-03 Wave 1):**

Sub-G3a — Synchronous teardown + activation token:

- `src/app/runtime/output-receiver/output-align-mode-loader.js`:
  - Module-level `let _activationToken = 0;` declared alongside
    `_currentBootHandle`.
  - `activate()` captures `const myToken = ++_activationToken;` at
    entry; 7 stale-token guards (`if (myToken !== _activationToken)
    return;`) inserted after each await.
  - `onProjectionProfileChange` handler now opens with synchronous
    `_currentBootHandle.stop?.()` AND
    `document.body.classList.remove("align-mode-active")` BEFORE any
    await. The sync `stop()` reuses Phase 38 W11's ordered-teardown
    path (HANDLE_UI.onAlignModeChange(false) BEFORE unsubscribe), so
    structural W11 invariant preserved.
- Commit `377f684`.

Sub-G3b — captureCurrentAsLoadedSnapshot helper:

- `src/app/runtime/viewport/runtime-projection-profile-persistence.js`:
  New helper:
  ```js
  function captureCurrentAsLoadedSnapshot(name) {
    if (!_gridStateApi) return;
    _loadedProfileName = name || null;
    _loadedProfileSnapshot = _gridStateApi.snapshotGridState();
    try { _persistLoadedProfileToLs?.(); } catch (_e) {}
    _recomputeAndNotifyDirty();
  }
  ```
  Exported via `window.TT_BEAMER_RUNTIME_PROJECTION_PROFILE_PERSISTENCE`.
- `src/app/runtime/output-receiver/output-live-sync.js`:
  `_applyAlignGridSnapshot` now invokes `captureCurrentAsLoadedSnapshot`
  right after `restoreGridSnapshot`, with a try/catch + lookup-at-call
  time pattern that handles the load-order race (output-live-sync wired
  before the lazy IIFE bundle finishes parsing).
- Commit `229b258`.

G3-PROBE diagnostic (39.1-01 Task 5) is permanently in place at
`runtime-projection-profile-persistence.js:617-646` and will fire on

## Carry-Forwards LOCKED for Future Phases

Three new patterns become first-class regression gates for future SSR
work:

- **Ultra-high preset + bitrate plumbing semantics.**
  `deriveSimulcastBitrates({ effectiveStreamFpsCap, encoderConfig })`
  with `Math.max(env, configOverride, fpsDerived)` semantics. Env
  still wins (operator-debug escape hatch); config can RAISE bitrate
  but never silently lower it below the fps-derived default — the
  "high+" interpretation.

- **Per-triangle position dilation pattern.** flat-arrays drawArrays
  + DILATE_PX outward shift along centroid→vertex unit vector in NDC
  space. Adjacent triangles' dilated regions overlap at their shared
  edge with identical UVs — invisible overdraw eliminates rasterizer
  diamond-exit ambiguity. Costs 6× vertex submission at interior
  grid vertices (still O(tens of kilobytes) at 81-vertex grids).

- **Synchronous teardown + activation-token pattern for async
  re-mount.** Async re-mount sequences MUST tear down prior state
  synchronously BEFORE any await; track an activation token; stale
  activations exit early on token mismatch.

- **Authority-pushed-snapshot pattern
  (`captureCurrentAsLoadedSnapshot`).** When a consumer module
  receives a snapshot from the authoritative source, it should
  re-baseline its local "dirty-against-snapshot" state immediately.
  Otherwise the local dirty flag becomes stale relative to the
  actual server state.

## Anti-Pattern Reinforced

**Anti-pattern: Using a JPEG-CDP screenshot as a verification gate for
an H.264-stream behavior.**

This is the operator-flagged anti-pattern from the Phase 39 UAT
2026-05-13. Phase 39's D-03 test
(`test_phase39_d03_no_seams.py`) measured pixels via CDP
`Page.captureScreenshot` (JPEG quality=60), reported `max_delta=0`, and
Phase 39 closed automated — but the operator's actual stream is
`getDisplayMedia → H.264 baseline → WebRTC → Pi decode`, where 4×4
transform blocks aligned to mesh-cell edges quantize the same
single-pixel ridge into visible structured noise. Same source image,
different compression artifact pattern, different visibility threshold.

Net effect: a green test for a fix that didn't address the operator's
actual problem.

**Added to `.planning/CRITICAL_KNOWN_BUGS.md` as entry #6**: future
test discipline for ANY change touching the SSR rendering or encoding
pipeline MUST capture pixels from the `<video>` element via
canvas.drawImage + getImageData, NOT from a CDP screenshot. JPEG
screenshots remain valid for verifying the IN-PAGE canvas
(pre-encode); they are NOT valid for verifying the consumer-side
decoded stream.
