# Plan 21-1: Hull-Flicker breaks Solid-Color (per-room, opt-in)

## Objective

Make the `hull-flicker` room animation able to "switch off" any concurrent
`solid-color` room animation in the *same room* during its flicker off-periods.
This turns solid-color from a "blink over a lit lamp" into a real broken-lamp
effect: the light actually goes dark when hull-flicker's gate is closed.

Strictly scoped:
- Opt-in only — default off, current behaviour unchanged.
- Flag lives on a single coded backbone (`hull-flicker`).
- Interaction only affects `solid-color` room animations. Every other type
  (gif, mp4, other coded) is untouched.
- Per-room coupling: a hull-flicker in room A has no effect on solid-color
  running in room B. Cluster-triggered hull-flicker acts per member-room.

## Context

Room animations live under each board's `roomFxByBoard[boardId].animations`.
`hull-flicker` is registered both as an Inside-global animation *and* as a
room coded effect (`getRoomCodedAssetKeys()` in `runtime-asset-refs.js:39`).

The draw path for a room coded effect is `drawRoomComposition` →
`drawEffectVisual(type, ...)`. The hull-flicker branch in
`runtime-effect-visuals.js:117` already computes an `isOnPeriod` / `dipAlpha`
gate from a `flickerNoise(step * 0.08 + 3.9) > 0.72` test. That gate drives
its own black-dim + yellow-tube overlay today.

The gate is deterministic in `(age, speed, intensity)` — age is seconds
since the animation started, scaled by `state.animationSpeed` and per-animation
speed. So we can compute the same gate from any caller that knows a running
hull-flicker animation's `startedAt` + `speed`.

`solid-color` is its own branch at `runtime-effect-visuals.js:209`. It fills
the room's polygon bbox with a solid RGBA.

## Approach

1. **Add a persisted flag** `breaksSolidColor: false` to the room animation
   definition normalizer. Only meaningful for `assetRef === "hull-flicker"`;
   harmless noise on other defs. Round-trips through live-sync automatically
   once the normalizer knows about it.

2. **Extract the flicker gate** into a pure helper
   `isHullFlickerLampOff(age, speedHint = 1, intensityHint = 1)` in
   `runtime-effect-visuals.js`. Expose it on the module's public export
   object and thread it through `ctx`. The existing hull-flicker branch
   calls the helper so the visual and the gate stay in sync (no second
   noise source).

   Definition: returns `true` when the lamp should be visually off for
   this frame (i.e. `!isOnPeriod` — the gate is closed). Dips during
   an on-period count as on (the lamp is dimmed, not off). Signature keeps
   options optional so non-matching call sites don't have to know about
   intensity.

3. **Gate solid-color in the room composition path**. In
   `drawRoomComposition`, when `assetType === "coded"` and
   `resolvedEffectType === "solid-color"`:
   - Look up any running animation on this board + roomId whose effective
     coded asset ref resolves to `"hull-flicker"` AND whose definition has
     `breaksSolidColor === true`.
   - Cluster-scoped animations count: when iterating, flatten cluster-run
     members (they each carry a `roomId`).
   - If one is found and `isHullFlickerLampOff(sibling.age, sibling.speed)`
     is true → return without drawing the fill. Otherwise draw normally.

   A helper `findActiveHullFlickerGate(boardId, roomId, now)` on the draw-loop
   module encapsulates the lookup so the branch stays readable. The helper
   returns `{ age, speed }` for the first matching sibling, or `null`.

4. **UI**. In `index.html`, add a new `<label id="room-breaks-solid-color-label">`
   next to the solid-color colour picker, containing a single checkbox.
   Wire it in `runtime-wire-room-audio-binders.js` following the
   `roomColorPicker` pattern:
   - Add `roomBreaksSolidColorInput` + `roomBreaksSolidColorLabel` to
     `runtime-dom-refs.js`.
   - New `syncBreaksSolidColorVisibility(definition)` shown only when
     `assetRef === "hull-flicker"`.
   - Checkbox `change` → update draft → same save path as other
     per-definition tweaks (intensity/speed/opacity/colorHex). Follow whatever
     existing field already writes back to the definition on the current
     selected animation — the flag rides along with the same patch.
   - On definition-dropdown change, `applyDefinitionDefaultsToDraft` copies
     `breaksSolidColor` into the draft so the checkbox reflects the saved
     value.

5. **Styles**. Minor — make the label stay inline with the existing
   solid-color colour-picker block. Only add CSS if the default layout
   reads wrong.

## Non-goals

- Inside-global hull-flicker (ship-wide dim + tint) gets no new behaviour.
- No cross-room interaction (A's hull-flicker doesn't touch B's solid-color).
- No interaction with any animation type other than solid-color.
- No config UI on the Inside hull-flicker definition — flag is only surfaced
  on the Room animation definition where it applies.

## Must-haves

- Default off preserves current behaviour pixel-for-pixel.
- When on, solid-color goes dark during hull-flicker's off-gate frames.
- Helper reuse — the hull-flicker visual and the solid-color gate read from
  the exact same `isHullFlickerLampOff` function (no drift).
- Cluster-triggered hull-flicker (e.g. "all rooms" cluster) gates solid-color
  in every member room, independently.
- Flag persists through board profile save/load and through live-sync
  across clients.
- Only activates when a hull-flicker animation is actually *running* on that
  room — a saved flag on a stopped definition has no render-time effect.

## Tasks

### T1 — Persist `breaksSolidColor` on room animation definitions
- Add the field to `normalizeRoomAnimationDefinition` with `Boolean(...)`
  coercion and default `false`.
- Verify the existing profile save/load and live-sync plumbing carries it
  (should be free if the normalizer knows it).

### T2 — Extract `isHullFlickerLampOff` helper
- Pull the `timeline → step → gate` computation out of the hull-flicker
  branch in `runtime-effect-visuals.js` into a module-level function.
- Export it on `window.TT_BEAMER_RUNTIME_EFFECT_VISUALS`.
- Thread it into draw-loop ctx via `runtime-orchestration.js` (if that's
  where effect-visuals is wired — verify; otherwise wire in whichever
  orchestration file already passes `drawEffectVisual`).
- Rewrite the hull-flicker branch to call the helper for its own
  `isOnPeriod` check — zero behaviour drift.

### T3 — Gate solid-color in the draw path
- In `drawRoomComposition`, after resolving coded effect type, and before
  calling `drawEffectVisual` for `"solid-color"`, consult
  `findActiveHullFlickerGate(boardId, room.id, now)`.
- If it returns a gate and `isHullFlickerLampOff(age, speed)` is true → skip
  the fill (return / no-op).
- Implement `findActiveHullFlickerGate` to scan `state.runningAnimations`:
  - For `scope === "room"` entries on the same board+room, read the running
    animation's `type` → look up the definition on the room profile → check
    if its resolved coded asset ref is `"hull-flicker"` and
    `breaksSolidColor` is true.
  - For `scope === "cluster"` entries on the same board, walk the member
    runtime views (`buildClusterMemberRuntimeViews`) and apply the same
    check per-member for `memberView.roomId === room.id`.
  - Return the first match's `{ age, speed }` computed from its `startedAt`
    and `state.animationSpeed`.

### T4 — UI: checkbox in the Room animation editor
- `index.html`: add `<label id="room-breaks-solid-color-label" style="display:none">`
  with a checkbox `id="room-breaks-solid-color"` next to
  `#room-color-picker-label`. Label text: "Break connected solid-color lamps"
  (exact wording TBD — keep concise).
- `runtime-dom-refs.js`: add `roomBreaksSolidColorInput` /
  `roomBreaksSolidColorLabel` refs.
- `runtime-wire-room-audio-binders.js`: mirror the `colorHex` wiring —
  show/hide on definition change, push to draft on input, include in the

- `applyDefinitionDefaultsToDraft`: copy `breaksSolidColor` from def → draft.
