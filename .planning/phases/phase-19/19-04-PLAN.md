# Plan 19-4: Unified Grid Projection System

## Objective
Replace the broken dual-system (4-corner CSS matrix3d + offscreen canvas grid mesh) with ONE unified grid-based projection system. The grid IS the projection — corners, edges, and interior points are all part of the same grid. Default 4x4. Right-click to add/remove lines.

## Problems with Current Implementation
1. Two separate UI systems (red 4-corner handles + green grid handles) that don't interact
2. Offscreen canvas approach causes animation freeze when grid points are moved
3. No visible deformation when adjusting grid points
4. Grid lines not draggable as whole units

## New Architecture
- ONE grid of control points covering the entire output viewport
- Default 4x4 grid = 5x5 = 25 control points (including edges and corners)
- The 4 corner points replace the old 4-corner system
- Edge points move along their axis only (like old edge handles)
- Interior points move freely in both axes
- Grid lines between adjacent points draggable as a whole (row or column)
- Right-click to add/remove horizontal/vertical lines
- All stored in localStorage as one unified state

## Rendering Approach
- Remove the offscreen canvas approach entirely (too slow, too buggy)
- Use CSS matrix3d on the .stage element computed from the 4 CORNER points of the grid (same as before)
- For interior grid distortion: apply per-cell affine transforms by subdividing the canvas drawImage into grid cells ONLY during the final copy step, not as offscreen pre-render
- Simpler: render normally to canvas, then in endGridWarpFrame just re-draw from canvas to itself with per-cell mapping using a temp ImageBitmap snapshot

Actually simplest approach that works:
- Render animations normally to the main canvas (no offscreen swap)
- After draw completes, if grid has non-zero interior displacements, take a snapshot (getImageData or createImageBitmap) and redraw through the grid mesh
- This happens once per frame, after all rendering is done
- Performance: only active when grid is adjusted (most of the time it's a no-op)

## Tasks

### T13 — Rewrite projection-mapping.js as unified grid system
- Remove all 4-corner-specific code (corners state, corner handles, edge handles)
- Remove offscreen canvas code (beginGridWarpFrame/endGridWarpFrame)
- New unified state: `{ gridX: [0, 0.25, 0.5, 0.75, 1], gridY: [0, 0.25, 0.5, 0.75, 1], points: { "row-col": { x, y } } }`
  - gridX/gridY define the grid line positions (normalized 0-1)
  - points stores the displaced position of each grid intersection
  - Default: points[row][col] = { x: gridX[col], y: gridY[row] }
- CSS matrix3d computed from the 4 corner points (grid corners)
- Post-draw mesh warp from interior point displacements

### T14 — Unified handle UI
- Single set of handles: circles at each grid intersection
- Corner handles (4): red, larger, labeled 1-4
- Edge handles: blue, medium
- Interior handles: green/cyan, smaller
- All draggable with same drag system
- Corner: free movement (defines perspective)
- Edge: constrained to axis
- Interior: free movement (defines local distortion)
- Grid lines between handles: draggable as whole row/column

### T15 — Post-draw mesh warp
- After draw() completes, check if any interior points are displaced
- If yes: snapshot the canvas, clear it, redraw through mesh grid
- Use createImageBitmap for async snapshot (non-blocking) or getImageData as fallback
- For each cell: drawImage(snapshot, srcRect, dstRect)
- Source rects: original evenly-spaced grid
- Dest rects: displaced grid positions
- Only on /output, only when displacements exist

### T16 — Cleanup draw loop
- Remove beginGridWarpFrame/endGridWarpFrame hooks from draw loop
- Add a single postDrawMeshWarp() call after the main draw
- No canvas swapping, no ctx mutation

## Must-haves
- One unified grid with corners, edges, interior points
- 4 corner points control CSS perspective (like before)
- Interior points provide local distortion
- Grid lines draggable as whole rows/columns
- Right-click to add/remove lines
- No animation freeze
- Visible deformation when points are moved
- localStorage persistence
- No performance impact when grid is at default positions
