Fix mechanism (Direction B, the default):

In `src/app/runtime/viewport/runtime-stage-viewport.js`, function `setAlignMode`,
the dashboard-emit branch currently emits and waits for the server snapshot to
echo back the new alignMode before updating the local panel. Change that to:

  1. ALSO update `state.alignMode = nextAlignMode` synchronously, before the
     async emit fires.
  2. Call `syncAlignModePanel()` synchronously, so the icon-button aria-pressed
     + the indicator bar + the body class `.align-mode-active` all flip on the
     same frame as the click.
  3. Call `syncAlignModeDirtyDashboardState()` synchronously (it re-reads
     `state.alignModeDirtyOnOutput` which is already authoritative-from-server,
     so this just refreshes the chip + button-disabled attribute without
     changing the underlying flag).
  4. Call `ctx.renderRoomOverlay()` synchronously so the overlay matches the
     new state.
  5. KEEP the existing emit + pending-snapshot feedback message — the server
     round-trip still happens; we just don't WAIT for it to update the UI.
  6. ROLLBACK on emit failure: in the `.catch()` branch, revert state.alignMode
     to the previous value and re-sync the panel (the existing
     "Status: align-mode command failed" message already exists; we just need to
     also revert state).

Why this fixes the hiccup:
  - Pre-fix: state.alignMode flips only when applyLiveRuntimeSnapshot's hoist
    block (line ~374) runs after the next poll cycle (120–250 ms). During that
    window, ANY render path that gates on state.alignMode (the indicator bar
    via syncAlignModePanel, the chip via syncAlignModeDirtyDashboardState)
    shows stale "ON" state.
  - Post-fix: state.alignMode flips on the click frame, so all UI consumers see
    the clean state immediately. The server's eventual snapshot apply is
    idempotent (the existing `_lastAlignModeState` gate at stage-viewport.js
    line 146 already prevents double-firing onAlignModeChanged).

The running-animations list hiccup is a SEPARATE consumer-side effect: when
applyLiveRuntimeSnapshot runs, it calls `replaceChildren()` then re-builds
rows. If the dashboard render queue is briefly blocked (e.g. by the
align-mode-active body class' CSS animation `align-glow-pulse` from styles.css
line 1200), removing the body class earlier (step 2 above) lets the browser
recompute layout/paint sooner. This is why Direction B alone is expected to
also collapse the "no running animations" gap — both symptoms share the same
root cause (stale state.alignMode delaying the body class flip).

# Target shape post-Wave 2:

```js
  function setAlignMode(enabled, { emit = true } = {}) {
    const state = ctx.state;
    const nextAlignMode = Boolean(enabled);
    if (emit && ctx.outputRole === ctx.OUTPUT_ROLE_CONTROL) {
      // Phase 48 W2 — optimistic dashboard update. Pre-fix, state.alignMode
      // only flipped after the server's snapshot echo (120-250 ms later),
      // leaving the dashboard's dirty-chip + indicator + body class stale.
      // Now we mutate locally + sync panels synchronously, then emit + await
      // server confirmation. On emit failure we rollback to the previous state.
      const previousAlignMode = state.alignMode;
      state.alignMode = nextAlignMode;
      syncAlignModePanel();
      ctx.renderRoomOverlay();
      void ctx.emitLiveMutation("context-update", {
        reason: "align-toggle",
        alignMode: nextAlignMode,
        runtime: { alignMode: nextAlignMode },
      }).then(() => {
        ctx.triggerFeedback.textContent = `Pending: align mode ${nextAlignMode ? "ON" : "OFF"} (waiting for snapshot)`;
      }).catch(() => {
        // Phase 48 W2 — rollback on emit failure.
        state.alignMode = previousAlignMode;
        syncAlignModePanel();
        ctx.renderRoomOverlay();
        ctx.triggerFeedback.textContent = "Status: align-mode command failed";
      });
      return;
    }
    state.alignMode = nextAlignMode;
    syncAlignModePanel();
    ctx.renderRoomOverlay();
  }
```

Note: we do NOT optimistically force `state.alignModeDirtyOnOutput = false`.
That flag is server-authoritative (only /output/ writes it via the dirty-POST
broadcaster, see runtime-projection-profile-persistence.js:294-298). We just
re-sync the chip/button-disabled with whatever the CURRENT server-known dirty
value is — syncAlignModePanel internally calls syncAlignModeDirtyDashboardState
on the CONTROL role (line 155 of stage-viewport.js), so step 2 above
transitively refreshes the chip too.

# Why this doesn't break /output/:

The whole new code path is inside `if (emit && ctx.outputRole === ctx.OUTPUT_ROLE_CONTROL)`.
/output/ takes the `else`-equivalent path (the final 3 lines of the function),
which is unchanged. /output/ continues to mutate state.alignMode + sync
synchronously the same way it always did.

# Why this doesn't break the server-echo path:

applyLiveRuntimeSnapshot's alignMode hoist (live-sync-core.js line 374-389)
sets state.alignMode based on the snapshot then calls syncAlignModePanel().
syncAlignModePanel has the `_lastAlignModeState` gate (stage-viewport.js
line 146) so calling it when state.alignMode hasn't changed is a CSS toggle
no-op + skips onAlignModeChanged. Post-Wave-2: our optimistic update sets
state.alignMode + calls syncAlignModePanel → _lastAlignModeState becomes the
new value. When applyLiveRuntimeSnapshot echoes the same value 120-250 ms
later → _lastAlignModeState already matches → onAlignModeChanged is a no-op.
Idempotent.
